Lesson 4.1: Introduction to Networks

Networks

A network is a group of entities that can communicate, even though they are not all directly connected.

To make a network, we need the following:

  • Way to encode and interpret messages
    Internet: message > bits > electrons/photons
  • Way to route messages
    Internet: routers figure out next hops
  • Rules for deciding who gets to use resources
    Internet: best effort service

Measuring Networks

Latency is the time it takes a message to get from a source to a destination. It is a unit of time, most likely measured in milliseconds (1,000 milliseconds in 1 second).

Bandwidth is the amount of information that can be transmitted per unit of time. It is measured in units of information divided by units of time, for example in bits per second.

A bit is the smallest unit of information. In programming, we think of bits – yes and no responses of information – as 0s and 1s, whereas 0 means no and 1 means yes. We can transmit anything we want by using bits.

Protocols

A protocol is a set of rules that people agree to that tell you how two entities can talk to each other. For the web, the protocol gives rules about how a client (web browser) and a server talk to each other (web server) – HTTP (Hypertext Transfer Protocol).

Query Parameters (GET parameters)

Introduced by a ? and followed by the parameter with its name and value being what appears after the question mark. We can introduce more query parameters by adding an & after the initial query parameter.

HTTP Requests

When your browser makes a request to a web server, the first thing it sends is the request line, which has three main parts:

  • Method: what type of request we're making (GET and POST are the most popular methods)
  • Path: the actual document we're requesting from the server
  • Version: always HTTP/[version number] (we'll be dealing mostly with version 1.1, and sometimes with 1.0)

The next thing your browser sends to a web server when making a request are a series of Headers. Headers always include the host name (Host: [host-name]) and a user agent (user-agent: [web browser + version]), and can include anything else you want that will improve meta data and communication between computers.

HTTP Responses

Web servers respond to HTTP requests with a status line, which looks like HTTP/1.1 200 Ok, where "200" is a status code and "Ok" is a reason phrase.

Common Status Codes:

  • 200 (Ok)
  • 302 (Found)
  • 404 (Not found)
  • 500 (Server Error)

Status lines are followed by a series of Headers. Date is always returned, and can also include Server (not a good idea to include, though), Content-Type and Content-Length for example.

Servers

The purpose of a server is to respond to HTTP requests. There are two main classifications for the type of response a server will do: static and dynamic.

Static requests are pre-written files or images that are returned by the server. Dynamic requests are built dynamically on the fly, and are essentially web applications.

Lesson 4.4: Modulus and Dictionaries

Modulus Operator

This is an operator (like +, -, *, or /) that every programming language has. We use the modulus operator to get our output in the correct range, written with a %. What modulus does is take a number and map it to a range based on the remainder when you divide that number: [number] % [modulus] —> [remainder]. So, 14%12—>2. This video does a good job of explaining the modulus operator.

Dictionaries

A dictionary is a data structure with key:value pairs, where we have a series of keys and an associated value for each key. It is created using {}. Dictionaries are mutable, meaning we can always change its contents. The values in a dictionary can be anything, even other dictionaries. This video gives an example of values in a dictionary being other dictionaries, and what you can do with that.

Lesson 4.5: Working with Google App Engine

Get vs. Post

Differences between get parameters and post parameters include the following:

Get:

  • Parameters are included in the URL
  • Used for fetching documents
  • Maximum URL length
  • Ok to cache
  • Shouldn’t change the server

Post:

  • Parameters are included in the body
  • Used for updating data (making changes to data stored on a server)
  • No maximum URL length
  • Not ok to cache
  • Ok to change the server

Lesson 4.6: Validation

Validation

Validation basically means verifying on the server side that what we received is what we expected to receive. Your server can receive junk, so it needs to be smart enough to deal with somebody sending input/parameters to it that can break it down or exploit it.

The steps we need to take in validation include:

  1. verify the user’s input
  2. on error, render form again
  3. include error message

Validation can also improve the user experience by providing feedback to the user, normally in the form of an error message when invalid input is given. This feedback can help the average user by telling them what type of input they are expected to provide.

HTML Escaping

HTML allows you to “escape” certain special characters, or change them. If we want our HTML to behave like HTML, then we don’t escape it. If we want to display HTML code but not have it behave like HTML, then we need to escape it. It’s always best practice to use built-in modules for escaping HTML.

Lesson 4.7: HTML Templates

Templates

Templates generally refer to a library, where a template library is a library to build complicated strings (which normally refers to HTML). Templates allow us to:

  • Separate different types of code
  • Make more readable code
  • Make more secure websites
  • Make HTML that is easier to modify
  • Avoid repetition of repeating elements in our code; this is helpful because it simplifies code, eliminates unnecessary work and helps us avoid silly errors

Jinja is a templating language (jinja2 template library), which is built into Google App Engine. More information about jinja2 can be found at ninja.pocoo.org.

Helpful Tips

  • Always automatically escape variables when possible (via ‘autoescape = True’ in jinja, for example)
  • Minimize code in templates
  • Minimize HTML in code

Lesson 4.8: Databases

Databases

Databases are where you actually store all of this information you collect from users, and is the final piece that allows us to build a fully functional web app. Specifically, a database is a program that stores and retrieves (large amounts of structured) data.

Tables

In general, databases are made up of a series of tables that contain large amounts of structured data.

Why Databases?

Databases exist to take large chunks of data and answer queries on them in a reasonable amount of time without having to write a lot of custom code.

Google App Engine Datastore

In Google App Engine Datastore, tables are called entities. In entities, columns are not fixed, all entities have an ID, and entities have parents/ancestors (which help to get around some of the problems with consistency).
